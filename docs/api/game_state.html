game_state

The game state machine requires a number of components to be useful:

  • GameState: An enumeration of all the possible game states. This enumeration is used by the main event loop to keep track of which state handler to invoke, as well as by the state handlers themselves to signal state transitions.
  • StateHandlerContext: A simple data object that stores the game context, which is passed to each state handler function when it is being invoked.
  • StateHandler: A simple interface that implements a state handler. State handlers are expected to adhere to this interface, and may implement any or no functions. Note that they should always call their parent functions listed in the interface, because these parent functions perform useful logic.

 41class StateHandlerContext:
 42    """
 43    The state handler context is populated by the main state machine event
 44    loop, and is provided to state handlers so that they have easy access to
 45    everything necessary to draw to the screen, process user input, and update
 46    the game state.
 47
 48    The state machine is run iteratively at a fixed frame rate. A
 49    `StateHandlerContext` gets generated for each and every frame that is
 50    rendered. During each iteration, all events are collected and passed into
 51    the context so that handlers can handle any events that may have passed
 52    since the handler function was last executed.
 53    """
 54    def __init__(self,
 55                 state: GameState, events: list[pygame.event.Event] | None,
 56                 window: pygame.Surface,
 57                 gui: pygame_gui.UIManager,
 58                 delta: float):
 59        self._state = state
 60        self._events = events
 61        self._window = window
 62        self._gui = gui
 63        self._delta = delta
 64
 65    def get_state(self) -> GameState:
 66        """
 67        Get the current game state. Note that this is not always the state
 68        that the currently running state handler was registered to handle. See
 69        `StateHandler` for more details.
 70        """
 71        return self._state
 72
 73    def get_events(self) -> list[pygame.event.Event] | None:
 74        """
 75        Get a list of Pygame events that have taken place since the last
 76        iteration. Each fired event only shows up in this list exactly once,
 77        so it is recommended to iterate over it during each function call and
 78        process the events that need to be dealt with as soon as they are
 79        fired.
 80        """
 81        return self._events
 82
 83    def get_window(self) -> pygame.Surface:
 84        """
 85        Get the Pygame drawing surface onto which all game components are
 86        blitted. This surface represents the entire screen area, so each state
 87        handler can draw anywhere on the screen and update the entire screen
 88        as necessary. Note that individual handlers don't have to update the
 89        display by calling out to pygame explicitly; this is all handled by the
 90        state machine event loop. State handlers must only blit everything onto
 91        this surface.
 92
 93        Note that the window is not preserved across frames; each state handler
 94        will have to re-draw the entire frame on every iteration of the state
 95        machine. By default, each frame is filled in solid black, so if there
 96        are black parts on the window that a state handler did not draw, it is
 97        simply because the state handler did not blit anything over the empty
 98        frame in that location.
 99        """
100        return self._window
101
102    def get_gui(self) -> pygame_gui.UIManager:
103        """
104        This game utilizes Pygame GUI for some user inputs. If a state handler
105        must register components, it should do so with this `UIManager`, which
106        is automatically managed and updated by the state machine loop.
107
108        If a state handler is going to register GUI components, it should do so
109        only once when the state is entered&mdash;not iteratively. Note that
110        by default, the `UIManager` is cleared automatically on every state
111        change.
112        """
113        return self._gui
114
115    def get_delta(self) -> float:
116        """
117        Get the amount of time that has passed (in *seconds*) since the last
118        invocation of a state handler. This is effectively the time since the
119        last frame was rendered, and should be used to compute object motion in
120        a frame rate-independent manner, because the frame rate may not always
121        be steady on all platforms.
122        """
123        return self._delta

126class StateHandler:
127    """
128    A `StateHandler` defines functions that are invoked at certain periods
129    during the lifetime of the state machine. State handlers are registered
130    with the main state machine loop so that they are executed when the game
131    enters a given state.
132
133    Each function takes a `StateHandlerContext` so that there are no
134    limitations as to what a state handler can do at any point during its
135    lifetime.
136    """
137    def __init__(self, context: StateHandlerContext):
138        """
139        Construct a new `StateHandler`.
140        """
141        pass

143    def on_enter(self, context: StateHandlerContext) -> None:
144        """
145        This function is invoked at the rising edge of a state transition. A
146        state transition occurs either when the main loop begins, or after a
147        previous state yielded its control of the state machine. It is invoked
148        *before* `process()` exactly once per state transition.
149
150        For our application, this function is primarily used to draw Pygame
151        GUI components onto the screen for the current state, since they only
152        need to be registered with the `UIManager` once.
153        """
154        pass
155
156    def process(self, context: StateHandlerContext) -> GameState:
157        """
158        This function is invoked iteratively at a fixed frame
159        rate&mdash;though this rate should be assumed to be
160        unknown&mdash;for the entire duration that the state machine remains
161        in the current state. It is always invoked after `on_enter()` and
162        before `on_exit()`, although it may be executed many times between each
163        of those invocations.
164
165        For our application, this function is primarily used to draw the game
166        scene and process user input.
167
168        This function should return the next state that the state machine
169        should enter. If the state machine should remain in the same state, it
170        should return `context.get_state()`.
171        """
172        pass
173
174    def on_exit(self, context: StateHandlerContext) -> None:
175        """
176        This function is invoked at the falling edge of a state transition. It
177        operates just like `on_enter()`, except it is executed when the current
178        state indicates that it would like to transition to another state. This
179        allows the state handler to clean up anything as necessary.
180
181        For our application, this function is primarily used to reset
182        Pygame GUI for each screen, as well as any state that should not carry
183        over the next time this state is entered.
184
185        By default, `on_exit()` will clear the Pygame GUI, so individual state
186        handlers need only register their components on enter; they don't have
187        to worry about de-registering them on exit. Note, of course, that this
188        function may be overridden and thus not execute if the child function
189        does not make a call to `super().on_exit()`.
190        """
191        gui_manager = context.get_gui()
192        gui_manager.clear_and_reset()
